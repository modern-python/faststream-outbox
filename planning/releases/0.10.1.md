# faststream-outbox 0.10.1 — actionable schema-drift errors

**Patch release.** One opt-in diagnostic refinement plus docs/branding. No
public-API change and no breaking change beyond 0.10.0's. The only change to the
installed package is `validate_schema()`'s error text (see
[Compatibility](#compatibility)); everything else is documentation.

## Diagnostics

- **`validate_schema()` now tells you how to fix Alembic-blind drift (#99).**
  Two kinds of drift it detects cannot be remediated by `alembic revision
  --autogenerate` — a missing/altered `<table>_lease_ck` CHECK (Alembic has no
  check-constraint comparator) and a drifted partial-index predicate (Alembic's
  index comparator ignores `postgresql_where`). Previously the raised
  `RuntimeError` was a dead end: re-running autogenerate produced an empty
  `upgrade()`. Now, **when one of those two probes fires**, the error appends a
  pointer to a new docs section —
  [`docs/operations/alembic.md`](https://faststream-outbox.modern-python.org/operations/alembic/#fixing-drift-autogenerate-cant-see)
  — that holds the exact hand-written `op.create_check_constraint(...)` /
  `op.drop_index(...)` + `op.create_index(..., postgresql_where=...)` recipes.
  Autogenerate-fixable drift (missing columns, plain indexes, the DLQ table)
  gets **no** pointer — for those you just re-run autogenerate.

## Compatibility

`validate_schema()` is **opt-in** (you call it from a health check / CI gate; it
is never run by `broker.start()`). If you don't call it, nothing changes.

If you do: the error message gains a trailing pointer block, but the prefix
(`"Outbox schema mismatch: "`) and every per-error string are **unchanged**, so
existing log greps and `pytest.raises(..., match=...)` assertions on the message
still hold. No other behavior change — producers, subscribers, the lease /
terminal-write paths, timers, and the `dlq_table=None` path are all identical to
0.10.0.
