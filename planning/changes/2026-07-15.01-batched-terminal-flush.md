# Design: Batched terminal flush

## Summary

The subscriber's terminal write is one `DELETE` round-trip per message, and with
the default `max_workers=1` those DELETEs are fully serialized — the benchmark
baseline measures exactly **1 terminal DELETE per message** and ~840 µs/message
spent purely in that round-trip on loopback, capping single-worker throughput at
~1200 msg/s regardless of handler speed.

This change adds an **opt-in** per-worker coalescer that accumulates completed
plain-delete rows and flushes them as a single
`DELETE … WHERE (id, acquired_token) IN (VALUES …) RETURNING id`, collapsing the
terminal round-trip from 1/message to 1/batch. It is gated by a new per-subscriber
knob `terminal_flush_batch_size` (default **1** = today's per-row path,
bit-for-bit identical). The retry `UPDATE` and the DLQ `DELETE…INSERT` CTE stay
per-row. It ships with a gated benchmark before/after proving the round-trip drop.

## Motivation

The performance review against Denis Tsvettsikh's *«50 оттенков Transactional
Outbox»* placed the library in his **Variant 2** and identified the per-row
terminal DELETE as the single largest throughput lever. The benchmark harness
(shipped in `2026-07-14.01`) then quantified it: prototyping measured a per-row
DELETE at ~840 µs/message versus a batched DELETE at ~7.7 µs/message — a **110×**
wall-clock difference — while WAL bytes barely moved (1.12×). The cost is
round-trips, not database work: each terminal DELETE is a network round-trip, and
at `max_workers=1` they serialize.

The committed baseline records the starting point this change must move:
`delete_calls == messages` (exactly one terminal DELETE per message).

## Design

### The batchable / inline split

`dispatch_one` today runs the handler, classifies the outcome, and flushes inline
per row. The split between what batches and what does not is defined by the
*shape of the terminal write*, which the current code already decides via one
predicate — whether a `dlq_payload` exists at flush time:

- **Batchable** — a terminal write that is a plain
  `DELETE WHERE id=? AND acquired_token=?` with **no** DLQ payload. This is the
  hot path: successful acks, and terminal-failures when no `dlq_table` is
  configured.
- **Inline (unchanged)** — the retry `UPDATE` (per-row delay / attempts /
  timestamps) and the DLQ `DELETE…INSERT` CTE (per-row failure context).

The rule is literally "batch iff `dlq_payload is None` at flush time," so whatever
the current code does for max-deliveries, DLQ, and retry is preserved by
construction — none of those paths are re-decided.

### The client method

One new method on `OutboxClient`, mirrored in `AbstractOutboxClient` and
`FakeOutboxClient` (the real-vs-fake parity contract):

```
delete_batch_with_lease(conn, pairs: list[tuple[int, UUID]]) -> set[int]
```

It issues a single statement and returns the set of ids actually deleted:

```sql
DELETE FROM <outbox>
WHERE (id, acquired_token) IN (VALUES (:id0, :tok0), (:id1, :tok1), …)
RETURNING id
```

The returned set **is** the per-row lease check: a row whose lease was reclaimed
mid-handler does not match, is absent from the set, and is treated as
`lease_lost`. This preserves the **lease-token invariant** exactly, with no new
mechanism. One round-trip, on the worker's existing `AUTOCOMMIT` connection,
atomic (all matching rows deleted, or the statement fails wholesale). The existing
per-row `delete_with_lease` stays for the DLQ path and the `batch_size=1` path.

### Per-worker coalescer and the worker loop

Approach: **per-worker buffer, opportunistic flush** — lock-free, reusing the
writer connection each worker already owns (no cross-worker coordination).

- `dispatch_one` stops flushing *batchable* rows inline. It runs the handler,
  classifies the outcome, and appends
  `(id, token, intended_outcome, metric_tags)` to the worker's buffer, then
  returns. Inline rows (retry, DLQ) flush and emit exactly as today.
- `_worker_inner` gains the flush trigger: drain immediately-available rows via
  `get_nowait()`, dispatching each into the buffer; flush when the buffer reaches
  `terminal_flush_batch_size` **or** the inflight queue is momentarily empty; then
  block on `await _inflight.get()`. Under load the buffer fills and flushes at
  `batch_size`; under a trickle the queue empties and it flushes immediately — no
  added latency, no timer.

### Metric timing

For batched rows, `dispatch_one` can no longer emit the `acked` / `nacked_terminal`
metric inline — it does not yet know whether the batched DELETE landed. The
flusher emits instead: after `delete_batch_with_lease` returns the deleted-id set,
it walks the buffer — id present → emit that row's intended metric; id absent →
emit `lease_lost` (phase `"terminal"`). This preserves the **P17** rule (a metric
fires only when that row's write actually landed) at batch granularity. Ids are
distinct primary keys, so the set-membership mapping is unambiguous.

### Drain and shutdown

A completed row sitting in the buffer (handler done, DELETE pending) redelivers if
the worker exits without flushing. So the worker must **flush its buffer on exit**,
graceful or not, before the connection is torn down. The existing custom `stop()` /
drain path (`drain.md`) is extended so each worker flushes as its final step. The
`dispatch_one` shutdown-race guard is unaffected: it fires *before* a row is
buffered (a row whose handler was skipped never enters the buffer). A failed
shutdown-time flush simply leaves those rows to lease-expiry redelivery.

### Config

One new per-subscriber knob `terminal_flush_batch_size: int = 1`, threaded through
`registrator.py`, `router.py`, and `fastapi/router.py` alongside the existing
subscriber kwargs, validated `>= 1` at registration in `subscriber/factory.py`
(the self-validating-config pattern). `1` = today's per-row path.

## Non-goals

- **No batching of retry or DLQ writes.** They carry per-row state (delay,
  attempts, timestamps; failure reason, exception) and are the exceptional paths,
  not the throughput bottleneck. YAGNI.
- **No default behavior change.** `terminal_flush_batch_size=1` is the default and
  is bit-for-bit identical to today. This is a second "off = identical" axis
  alongside `dlq_table=None`.
- **No cross-worker shared buffer and no timer-based flush.** Per-worker buffers
  already fill under load; the queue-idle trigger already bounds latency. Both
  alternatives add coordination or a task for a marginal gain.
- **No raw-asyncpg pipelining.** A single `DELETE … RETURNING` is one round-trip
  and stays in SQLAlchemy Core; pipelining independent per-row DELETEs is more
  invasive (bypasses Core) for a smaller, N-statement win.

## Semantics and risk

**The tradeoff, accepted deliberately:** batching accumulates completed rows before
the DELETE lands, so on an **ungraceful** crash (SIGKILL / OOM / power loss) up to
`batch_size` completed-but-undeleted rows redeliver. The outbox is already
at-least-once — handlers must already be idempotent — so this is not a new failure
*class*, only more of an existing one, and only on ungraceful exit (graceful
`stop()` flushes the buffer). It is opt-in and off by default, so no existing
deployment changes behavior on upgrade.

**Lease-hold window.** Buffered rows hold their leases until the batch flush, so a
worker holds up to `batch_size` completed-but-unflushed leases on top of the
existing `fetch_batch_size + max_workers` bound. Leases self-expire, so this is
release latency, not a correctness issue. `subscriber.md`'s lease-bound section is
updated to note it.

**Flush-failure atomicity.** The batched DELETE is one statement, so a connection
failure fails it wholesale — no partial batch. The buffered rows were never
deleted, their leases expire, they redeliver. This is simpler than the per-row
failure path, not riskier.

## Testing

TDD, integration-first. Each behavior gets a failing test before the code.

- **Parity:** `test_client_contract.py` gains `delete_batch_with_lease` on both the
  real client and the fake, pinning identical return-set semantics.
- **Lease-loss in a batch:** a buffer where one row was reclaimed mid-handler emits
  exactly one `lease_lost` + N−1 `acked`; `tup_del` reflects only the landed rows.
- **Mixed buffer:** acks and terminal-failures-without-DLQ in one flush emit the
  correct per-row metric each.
- **Idle flush:** a partial buffer flushes when the queue empties before reaching
  `batch_size`.
- **Drain:** a graceful `stop()` flushes a partial buffer — no redelivery.
- **Default identity:** `terminal_flush_batch_size=1` produces the same DB call
  counts as `main` (guarded by the benchmark, see below).
- **Test broker:** fake sync-mode dispatch semantics are unchanged.

## Benchmark validation

The payoff, measured against the committed baseline. The consumer sweep gains a
batched point `consumer/w1/b100/tfbs100` (`max_workers=1`, `fetch_batch_size=100`,
`terminal_flush_batch_size=100`) alongside its `tfbs=1` sibling `consumer/w1/b100`.
Realized numbers at 5000 messages:

- `delete_calls` drops from `messages` (5000, one DELETE per row) to `messages /
  tfbs` (**50**) at the batched point — the round-trip win, now a committed number.
  Wall-clock rose from ~430 to ~8400 msg/s (~20×) on loopback.
- `tup_del` stays at **5000** for both points — the same rows are deleted exactly
  once; only the round-trip count changes. `tup_hot_upd` stays 0.

**The gate, as shipped.** `tup_del` and the tuple counters are exact-gated
(structural correctness: every row deleted exactly once). `delete_calls` is
exact-gated for **all** points, batched included: Step 5 measured it at exactly 50
across 5 back-to-back runs with zero variance, because `fetch_batch_size ==
terminal_flush_batch_size` delivers full buffers and empty fetch polls never fire a
flush — the count is structural, not interleaving-dependent. (If a future config
split buffers — fetch not divisible by tfbs — this would move to a loose upper
bound; see the note at `EXACT_KEYS` in `benchmarks/report.py`.) The baseline is
regenerated in this PR and `just bench-check` gates the before/after so the
improvement cannot silently regress.
